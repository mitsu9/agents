# 週次計画スキル

マイルストーン駆動の週次計画を作成し、Linear Issue を整理する。

## タスク管理: Linear

タスクはLinearで管理する。Linear MCP を使ってIssueの読み取り・作成・更新を行う。

- **チーム**: Mitsu9
- **ステータス対応**: Backlog（未整理）→ Todo（今週コミット）→ In Progress（着手中）→ Done（完了）
- **ラベル**: 期限固定 / 戦略的 / 対応型（優先分類）、思考 / 実行（種別）、AI委任
- **Issue Description**: `## 完了条件` セクションに done_when、`## コンテキスト` セクションに補足を記載
- **Project**: Obsidian の projects/ と1:1対応

## 実行手順

### 1. 前週の振り返り

前週の週次ノート（`{goals-week}`）の「今週のマイルストーン」を確認し、達成状況をユーザーと振り返る。

未達成MSがあれば、原因と今週への影響を確認する。

#### タイムレビュー（サブエージェント委任）

time-reviewer サブエージェント（`.claude/agents/time-reviewer/`）を `period: week` で起動する。サブエージェントがカレンダー・Claude Code・Linearからデータを���集・分析し、構造化レポートを返却す���。メインスレッドのコンテキストには生データを載せない。

レポートから以下を振り返りの材料にする:
- 活動タイプ別・Area別の時間配分
- Project × 優先度の整合性
- MTG比率・思考時間の確保状況
- 改善提案

改善提案は、後続の Step 4（今週コミットするMS決定）および Step 7（MSからタスク分解）で今週の計画に反映する。例: 「思考時間が不足」→ 今週の思考スロットを確保、「優先度の高いPJに時間が取れていない」→ 今週のMS選定で優先する。

#### KPT

タイムレビューの結果も踏まえて、ユーザーと振り返り（KPT）を行い、週次ノートの「振り返り」セクションに記録する。

前週の週次ノートが存在しない場合はスキップする。

### 2. 完了タスクの転記

Linear で Done ステータスの Issue を取得する（`list_issues` で `assignee: me`, `state: Done`）。

完了タスクを前週の週次ノートの「完了タスク」セクションに転記する。

### 3. 全PJのMS一覧提示・{milestones} 更新

全プロジェクトファイル（`{projects}`）からマイルストーンを収集し、`{milestones}` を更新する:

- 週ごとのテーブル（今週のステータス反映、達成MSを「完了」セクションへ移動）
- 先頭の Mermaid ガントチャート（`done` / `active` の更新、新規MSの追加、週の移動）

ユーザーに「今週 (Www)」のマイルストーン候補を一覧で提示する。

### 4. 今週コミットするMS決定

ユーザーと相談して、今週コミットするマイルストーンを決定する:

- 思考型が複数ある場合、1つに絞る（優先度: 期限固定 > 戦略的 > 対応型）
- 実行型は全て入れる
- 戦略的PJの保護ルール: 直近2週で思考スロットがなかったPJを優先的に提案する

### 5. 月目標の確認

当月の月目標（`{goals-month}`）を読み、週の方向性と整合性を確認する。月目標ファイルが存在しない場合は、`/plan-month` での作成を提案する。

### 6. 週次ノートの作成

`templates/weekly-note.md` をベースに `{goals-week}` を作成する。決定したMSを「今週のマイルストーン」テーブルに記入する。

### 7. MSからタスク分解→Linear Issue 作成

決定したマイルストーンをタスクに分解し、ユーザーと相談して Linear Issue を作成する:

- 各タスクは Linear に Issue として作成する（`save_issue` で `team: Mitsu9`, `assignee: me`）
- AI委任タスクには `AI委任` ラベルを付与する
- Issue はPBIのように「何を達成するか」を定義する（サブタスクは持たない）
- Issue 作成時に以下を設定する:
	- `title`: タスク名（短く）
	- `project`: 対応する Linear Project 名
	- `state`: 今週取り組む → `Todo`、来週以降 → `Backlog`
	- `labels`: 優先分類（期限固定/戦略的/対応型）+ 種別（思考/実行）
	- `description`: 以下のフォーマットで記載

```markdown
## 完了条件
- [ ] 条件1
- [ ] 条件2

## コンテキスト
補足メモ
```

- `{thinking}/Documents/general/定例会議.md` を読み、曜日ごとの定例を把握し、週全体のキャパシティを考慮する
- 1週間に詰め込みすぎないよう注意する
- `{thinking}/Documents/general/思考キュー.md` に「今すぐ考えたい」項目があれば、思考セッションの時間確保を提案する

### 8. 既存 Backlog Issue のレビュー

Linear の Backlog Issue をユーザーと一緒にレビューする（`list_issues` で `assignee: me`, `state: Backlog`）。

今週取り組むべき Issue があれば、ステータスを **Todo** に昇格させる（`save_issue` で `state: Todo`）。
