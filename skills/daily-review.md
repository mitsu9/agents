# デイリーレビュースキル

終業時に振り返りを行い、翌日を設計するためのスキル。朝の `/daily` と対になる。

## タスク管理: Linear

- **チーム**: Mitsu9
- **ステータス**: Backlog → Todo → In Progress → Done
- **ラベル**: 期限固定 / 戦略的 / 対応型、思考 / 実行
- **Issue Description**: `## 完了条件` に done_when を記載

## フロー概要

```
1. 収集 & レポート（AI側で一括実行、ユーザー待ち時間なし）
2. WorkFlowy確認（あれば追加、なければスルー）
3. 確認（質問に答えるだけで進む形式）
4. 翌日設計
5. 振り返り（3問）
6. ノート更新 & バックアップ
```

## 実行手順

### 1. 収集 & レポート

以下をすべて並行で取得し、**1つのレポートとしてまとめて提示**する。ユーザーとのやり取りは発生しない。

**取得するもの:**
- Linear Issue一覧（`assignee: me`, Todo / In Progress）
- マイルストーン一覧（`{milestones}`）
- 当日のデイリーノート（`{daily-note}`）
- 当週のウィークリーノート（存在すれば）
- Claude Code 活動サマリー（`/activity-summary` の手順に従う）
- Notion 議事録（当日の `created_time` でフィルタ）
- 翌日のカレンダー（gws で取得。declinedを除外）

**レポートの構成:**

```
## 今日のレポート

### タスク消化
- 計画 vs 完了の対比（チェック状態）
- 完了条件を満たした Issue → Done に更新

### MS進捗
- 今週MSの状況を進捗率（%）付きで表示
- 週の経過割合（例: 火曜終了=40%）と対比して遅れを明示

### 決定事項・ネクスト
- Notion議事録 + Memoから抽出した決定事項とネクストアクション
- 議事録がないMTGの一覧（後で確認する）

### 活動サマリー
- セッション一覧と傾向（簡潔に）

### 明日の予定
- カレンダーからMTG一覧
- 空き時間の合計
```

### 2. WorkFlowy確認

「WorkFlowy に今日のメモはありますか？」と確認する。

- **ある場合**: 貼り付けてもらい、Memo セクションに転記。その後ステップ3へ
- **ない場合**: そのままステップ3へ

### 3. 確認（質問形式）

レポートをもとに、ユーザーが答えるだけで済む質問を投げる。一問一答ではなくまとめて聞く。

**聞くこと:**

1. **議事録なしMTGの決定事項・ネクスト**: 「以下のMTGで決定事項やネクストはありますか？（なければ『なし』で）」
2. **Obsidianへの追加**: 議事録・Memo・決定事項の中から、Obsidian（Documents）に追加すべき情報を提案する。特に以下は積極的に提案する:
	- チーム・人物に関する情報 → `{peoples}/`
	- 方針・戦略の決定 → `Documents/areas/` や ADR
	- プロセス・フローの変更 → 該当するarea配下
	- ClaudeがDocumentsを参照して議論する際に、この情報がないと文脈が欠けるものはすべて対象
3. **タスク化すべき項目**: ネクストアクションの中にLinear Issue化すべきものがあれば提案する
4. **ズレの確認**: 「レポートの内容にズレはありますか？」

回答を受けて、決定事項・ネクストの最終版を確定する。Linear Issue の作成・更新、Obsidian への追記が必要なものはこのステップで実行する。

### 4. 翌日設計

ステップ1で取得済みのカレンダー情報をもとに翌日を設計する。

**提示するもの:**
- 翌日のMTGタイムライン（空き時間を明示、ランチ30分を必ず確保）
- MTG準備が必要なものの指摘
- 今日の未完了タスクの扱い案（持ち越し / Backlog戻し / 完了扱い）
- 翌日のタスク案（Linear Todo Issue から選定、空き時間に合わせた現実的な量）

**聞くこと:**
- 「未完了タスクの扱いと、明日のタスクはこれでOK？」

### 5. 振り返り（毎日必ず実施）

3問構成。タスク消化（シングルループ）ではなく、自分自身の行動・判断を振り返る（ダブルループ）。

#### 共通の問い（毎日）

1. **リーダーシップモーメント**: 「今日、自分が目指すリーダー像に近い行動を1つ挙げるなら？」
	- 1文でOK。ネガティブ偏重を防ぎリーダーアイデンティティを強化する
2. **戦略整合チェック**: 「今日の時間配分は、最も重要な目標に対して適切だったか？」（YES/NO + 一言）
	- NOが続く場合は週次レビューで構造的な問題として扱う

#### 曜日別の問い

| 曜日 | レンズ | 問い |
|------|--------|------|
| 月 | **戦略** | 「今週の計画と今日の実態に、ギャップはあるか？ なぜ生じた？」 |
| 火 | **人** | 「今日チームの誰かについて気づいたことは？ 自分はその人にどう関われた？」 |
| 水 | **プロセス** | 「今日感じた摩擦・非効率は？ 何がもっとスムーズになり得る？」 |
| 木 | **学び** | 「今日の経験から、再利用可能な原則を1つ抜き出すとしたら？」 |
| 金 | **エネルギー** | 「今週、何にエネルギーを使い、何に消耗したか？ 自分の役割について何を示している？」 |

#### 進め方

1. 共通2問 + 曜日別1問をまとめて提示し、回答を受ける
2. 回答に対して、思考の盲点がないか1つだけ補完質問する（深追いしない）

### 6. ノート更新 & バックアップ

#### 当日デイリーディレクトリへのデータ保存

ステップ1で収集したデータを、当日のデイリーディレクトリ（`{daily}`）に個別ファイルとして保存する:

| ファイル | 内容 |
|----------|------|
| `activity-summary.md` | Claude Code 活動サマリー |
| `linear-snapshot.md` | Linear Issue のスナップショット（Todo / In Progress の一覧と状態） |
| `calendar.md` | 当日のカレンダー実績（実際に参加したMTG一覧） |
| `decisions.md` | ステップ3で確定した決定事項・ネクストアクション |

- ユーザーが手動で追加したファイルがディレクトリ内にあれば、そのまま残す
- 既存ファイルは上書きしてよい（日次で再生成するデータのため）

#### 当日デイリーノートの更新

当日のデイリーノート（`{daily-note}`）を更新する:

- **Log**: 完了タスク（詳細は `activity-summary.md` を参照）
- **決定事項・ネクスト**: ステップ3で確定した内容（詳細は `decisions.md` を参照）
- **振り返りメモ**: ステップ5の回答を記録
- **翌日設計**: ステップ4で確定した翌日タスク

既存の Memo セクションは変更しない。

#### 翌日デイリーノートの作成

翌日のデイリーディレクトリ（`{daily}` の翌日分）を作成し、`templates/daily.md` をベースにデイリーノートをほぼ完成形で作成する。

- **今週の目標・MS**: 週次ノートから転記
- **今日のプラン**: カレンダーからMTGタイムライン + 空き時間にタスク配置。ランチ30分を必ず確保する（MTGの隙間に【ランチ】ブロックを明示）
- **今日のタスク**: 選定したタスクを `#### MIT-xx: タスク名` で配置し、完了条件からアクションを展開
- **今週の残りタスク**: 翌日に選ばれなかった Todo Issue

翌日ノートが既に存在する場合は上書きせず、差分を提案する。

#### バックアップ

- `git add -A` → `git commit -m "Sync notes"` → `git push`

## 重要な原則

- **判断はユーザーに委ねる**: 未完了タスクの扱いを勝手に決めない
- **毎日振り返る**: 毎日3問の振り返りを必ず実施する。日報ではなく思考の整理
- **曜日で視点を変える**: 同じ問いの反復は思考が浅くなる。曜日別レンズで多角的に
- **完了条件で完了を判定する**: Linear Issue の `## 完了条件` を満たしているかで判断する
- **既存内容を破壊しない**: デイリーノートの Memo セクションは変更しない
