# プロジェクト起票スキル

新しいプロジェクトを起票し、初期検討に入るかどうかを意識的に判断するためのスキル。

## 実行手順

### 1. 起票（必ずやる）

ユーザーに以下の3点だけを確認する。簡潔に、5分以内を目安とする。

1. **プロジェクト名**: 何のプロジェクトか
2. **目的（1行）**: なぜやるのか
3. **優先分類**: 期限固定 / 戦略的 / 対応型

ユーザーが既に伝えてくれた情報は聞き直さない。不明な項目だけ聞く。

### 2. プロジェクトファイルの作成

`templates/project.md` をベースに `{projects}` にファイルを作成する。

記入する内容:
- プロジェクト名
- 目的（Step 1 で確認した1行）
- ゴール・アウトプット・関係者は空欄のまま残す
- プロジェクト情報（優先分類）を記入
- マイルストーンに「初期検討」を1つ仮置きする

```markdown
## プロジェクト情報
- 優先分類: [期限固定 / 戦略的 / 対応型]
- 紐付く目標: （未設定）
- 期限: [あれば記入、なければ空欄]

## マイルストーン

| # | マイルストーン | 種別 | 目標週 | ステータス |
|---|---------------|------|--------|-----------|
| 1 | 初期検討（ゴール・アウトプット・マイルストーンの具体化） | 思考 | - | - |
```

### 3. プロジェクト一覧・{milestones} の更新

`{projects-index}` のテーブルに新しいプロジェクトを追記する。概要には Step 1 で確認した目的を簡潔に記載する。

`{milestones}` に「初期検討」マイルストーンを追加する（週ごとのテーブルとガントチャートの両方）。

### 4. 判断ポイント — 今考えるか？

既存プロジェクトのマイルストーン状況を一覧で提示し、キャパシティの文脈を見せた上で、ユーザーに問う:

> 起票しました。初期検討（ゴール・アウトプット・マイルストーンの具体化）に今入りますか？
>
> - **今やる** → このまま検討に入ります
> - **後で** → 起票だけで終了します。`/plan-week` で適切な週に配置されます

#### キャパシティ文脈の提示方法

`{projects}` 配下の全プロジェクトファイルを読み、直近2-3週のマイルストーン状況を簡潔に表示する:

```
【直近のマイルストーン状況】
- W08: [思考] AI駆動開発 - カルテ機能PoCの進め方整理
- W08: [実行] FY26下期評価 - 評価シート提出
- W09: [思考] (空き)
- W09: [実行] AI駆動開発 - PRD・仕様書フォーマット定義
```

これにより「今この検討を入れる余裕があるか」をユーザーが判断できる。

### 5a. 今やる場合 — 初期検討

ユーザーと対話しながら以下を具体化する:

1. **ゴール**: どうなれば成功か
2. **アウトプット**: 具体的な成果物
3. **関係者**: 誰が関わるか
4. **マイルストーン**: 「初期検討」を具体的なマイルストーンに置き換える

プロジェクトファイルを更新する。`{milestones}` のテーブルとガントチャートも具体化したマイルストーンで更新する。

検討が深まりそうな場合は、ここから `/think-analyze` への接続を提案してもよい。ただし無理に誘導しない。

### 5b. 後でやる場合 — 起票だけで終了

「初期検討」マイルストーンが残った状態でプロジェクトファイルを保存して終了する。

次回の `/plan-week` 実行時に、このマイルストーンが週目標の候補として表示される。

## 重要な原則

- **起票は軽く**: Step 1-3 は素早く終わらせる。ここで深掘りしない
- **判断は意識的に**: Step 4 の問いを飛ばさない。これがこのスキルの核心
- **キャパシティを可視化する**: 判断材料なしに「今やる？」と聞いても意味がない。既存の状況を見せる
- **追加するなら何かを動かす**: DESIGN.md のルール「追加するだけは禁止」を遵守する。キャパ超過が明らかな場合は、何を遅らせるか確認する
