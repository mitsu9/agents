# ポートフォリオリバランススキル

プロジェクトポートフォリオを棚卸しし、「何を動かし、何をやらないか」を明示的に判断してマイルストーンを現実的に配置する。

PJは増える一方で減らない。このスキルの核心は「追加」ではなく「選択と集中」にある。やらないことを決める勇気を後押しし、有限なキャパシティの中で最大のインパクトを生む配置を一緒に考える。

## 基盤の読み込み

### 必須
- `context/shared.md`

### コンテキスト
- `{self-info}`: ミッション・役割
- `{challenges}`: 認識している課題
- `{thinking}/Documents/general/思考キュー.md`: 考えたいことのストック

### 目標（存在するものを読む）
- `Plan/YYYY-H1.md` または `Plan/YYYY-H2.md`: 当半期の目標
- `{goals-month}`: 当月の目標

### Area・PJ・タスク（すべて読む）
- `{areas}`: 全Areaファイル
- `{projects}`: 全プロジェクトファイル
- `{projects-index}`: プロジェクト一覧
- `{milestones}`: マイルストーン一覧
- `{thinking}/Tasks/self/`: 全タスクファイル（frontmatterを走査）

## 実行手順

全体の流れ: 現状把握 → PJ棚卸し → カバレッジ分析 → 優先度判断 → MS配置 → 反映

### Step 1: 現状の俯瞰

読み込んだ情報から、Area × PJ のマトリクスを提示する。

```
【Area × PJ 俯瞰】

■ アプリチーム
  🔴 FY27上期アプリチーム目標設定 — MTG3準備中、W14にMTG3
  🔵 Bチームスクラム導入支援 — MS未達、課題整理が滞留
  🔵 野原さん受け入れ — 委任済み、モニタリング
  🔵 田中さん受け入れ — 引き継ぎ中

■ デザインチーム
  🔴 FY27上期プロダクトデザインチーム目標設定 — 準備完了、W14にMTG1
  🔵 プロダクトデザインチーム体制検討 — 体制確定済み、残タスクあり

■ 評価・目標
  🔴 FY26下期評価 — 一次評価完了、FB面談4月
  🔴 FY27上期個人目標設定 — MGR相互レビュー完了、等級定義進行中

■ AI戦略・開発改善
  🟡 AI駆動開発 — PoC実施中
  🟡 事業部全体のAI活用戦略 — 進め方再検討で停滞

■ 運営
  🔵 端末管理 — MS未定のまま
  ⚪ 事務作業 — inbox大量滞留
```

凡例: 🔴期限固定 🟡戦略的 🔵対応型 ⚪非PJ

ユーザーに「この俯瞰で認識と合ってますか？状況変わっているPJがあれば教えてください」と確認する。

### Step 2: 既存PJの棚卸し

各PJについて、ユーザーと1つずつ判定を行う。判定は4択:

| 判定 | 意味 | アクション |
|------|------|-----------|
| **継続** | 引き続き動かす | MS・タスクの妥当性を後のStepで確認 |
| **完了** | ゴール達成済み | PJファイルに完了を記録、INDEX・Area更新 |
| **保留** | 今は動かさない（いつか再開） | MSをすべて「未定」に移動、Areaに保留と明記 |
| **中止** | もうやらない | PJファイルに中止理由を記録、INDEX・Area更新 |

判定の進め方:
1. Area ごとにまとめて提示する（Area の文脈の中でPJを見る方が判断しやすい）
2. 各PJについて、現状・残タスク・最後に動いた時期を簡潔に示す
3. 明らかに完了しているPJ（ゴール達成済み）は先にまとめて提示し、確認だけもらう
4. 判断に迷うPJはスキップして後で戻ってよい

保留にしたPJのうち「いつか再開する可能性がある」ものは `{thinking}/Documents/general/思考キュー.md` の「いつか考えたい」に追加する。

### Step 3: Area カバレッジ分析

棚卸し後の状態で、Area × ミッション のカバレッジを分析する。`/advise` のギャップ分析ロジック（Step 3: A. ミッション-活動のカバレッジ、B. 各プロジェクトの十分性、D. タイミングの妥当性）を適用する。

ただし `/advise` との違い:
- 外部情報（Slack/Notion/Calendar）は収集しない。cockpit内の情報のみで分析する
- 「新しい提案を生む」よりも「既存のArea に穴がないか」に焦点を当てる

具体的に確認すること:

#### A. Area の健康状態
- アクティブなPJがゼロの Area はないか
- 1つの Area にPJが集中しすぎていないか
- 半期目標に紐づくPJが存在しない目標項目はないか

#### B. 既存PJの十分性
- 継続と判定したPJのゴールに対して、MSは十分か
- 期限に対して進捗は現実的か
- 依存関係の見落としはないか

#### C. タイミング
- この時期に仕込んでおくべきことが漏れていないか（評価時期、期末、四半期の節目）
- 数ヶ月後に大きなイベントがあり、今から準備が必要なものはないか

分析結果を以下の形で提示する:

```
【カバレッジ分析】

✅ カバーされている
- [Area名]: [理由]

⚠️ 手薄になっている
- [Area名]: [何が足りないか、なぜ気になるか]

💡 検討候補
- [具体的な提案]: [背景]
```

提案は3個以内に絞る。多すぎると逆に判断が散る。

ユーザーと対話し、提案ごとに「やる / いつか / やらない」を判断する。「やる」の場合は `/project` で起票するか、既存PJにMSを追加するかを決める。

### Step 4: 優先度の判断

ここがこのスキルの最も重要なステップ。Step 2 で「継続」としたPJ + Step 3 で「やる」と決めたPJを並べて、優先度を決める。

PJが飽和しがちなユーザーにとって、「全部やる」は実質「全部中途半端になる」と同義。このステップでは、明示的に「今動かすPJ」と「意識的に後回しにするPJ」を分ける。

#### 優先度判断のフレーム

1. **必須（今動かさないと問題が起きる）**: 期限固定で直近にMSがあるPJ
2. **重要（半期目標の達成に直結する）**: 戦略的PJのうち、今動かさないと間に合わないもの
3. **望ましい（やれたらやる）**: 対応型、または戦略的だがもう少し後でもよいもの
4. **待機（意識的に後回し）**: 今は手を出さない。保留とは違い「いつ再開するか」を決める

提示の仕方:
```
【優先度判断案】

必須（今動かす）
1. FY26下期評価 — FB面談が4月中旬、準備が必要
2. FY27上期アプリチーム目標設定 — W14にMTG3

重要（並行して進める）
3. FY27上期プロダクトデザインチーム目標設定 — W14にMTG1
4. FY27上期個人目標設定 — 等級定義が前提

望ましい（余力があれば）
5. Bチームスクラム導入支援 — 課題整理は1回の時間確保で進む

待機（いつ再開するかを決める）
6. 事業部全体のAI活用戦略 — 再開目安: 5月以降
```

ユーザーと対話しながら調整する。特に「待機」に入れることへの抵抗がある場合は、「全部やろうとした場合にどうなるか」のシミュレーションを見せて判断を支援する。

### Step 5: マイルストーンの配置

優先度が決まったPJについて、マイルストーンを現実的に配置する。

#### 配置のルール
- DESIGN.md のルールに従う: 思考型は週1つまで、実行型は並列可
- `{thinking}/Documents/general/定例会議.md` を読み、各週の空きキャパシティを把握する
- 2週間先までは具体的に、それ以降は粗く配置する
- 「待機」PJのMSは「未定」セクションに移動し、再開目安日を添える

#### ユーザーとの確認
配置案をガントチャート風に提示し、ユーザーと調整する:

```
W14 (3/30〜)
  [思考] アプリ目標設定 — MTG3準備の最終確認
  [実行] アプリ目標設定 — MTG3実施
  [実行] デザイン目標設定 — MTG1実施

W15 (4/6〜)
  [思考] デザイン目標設定 — MGR擦り合わせ
  [実行] ...
```

### Step 6: 反映

ユーザーとの合意内容をファイルに反映する。

#### 更新するファイル
1. **`{projects}`**: 各PJのステータス・マイルストーン更新（完了/中止PJにはその旨を記録）
2. **`{projects-index}`**: PJ一覧の更新（完了PJの除外 or 完了マークの付与）
3. **`{milestones}`**: ガントチャート + 週テーブルの全面更新
4. **`{areas}`**: 各Areaの「現在のプロジェクト」「今どうなっているか」を更新
5. **`{thinking}/Tasks/self/`**: 中止/保留PJに紐づくタスクの整理（ユーザーに確認の上）
6. **`{thinking}/Documents/general/思考キュー.md`**: 保留PJ・「いつか」の提案を追加

#### タスクの整理
- 完了/中止PJに紐づく `status: inbox` のタスク → ユーザーに削除確認
- 保留PJに紐づくタスク → `status: inbox` のまま残す（再開時に使う）
- 継続PJの `status: inbox` タスク → 一覧提示し、this-week への昇格 or 維持を判断

### Step 7: セッションの保存

セッション全体を `{sessions}/YYYY-MM-DD_rebalance_portfolio-review.md` に保存する。

記録内容:
- 棚卸し前のPJ一覧（スナップショット）
- 各PJの判定結果（継続/完了/保留/中止 + 理由）
- カバレッジ分析の結果と判断
- 優先度判断の結果
- MS配置の決定内容
- 更新したファイルの一覧

## 他スキルとの連携

| 連携先 | タイミング | 内容 |
|--------|-----------|------|
| `/advise` | Step 3 | ギャップ分析ロジックを適用（外部情報収集はスキップ） |
| `/project` | Step 3 | 新PJを起票する場合 |
| `/plan-week` | 完了後 | リバランス結果を受けて週次計画を立て直す |

## 重要なポイント

- **「やらない」を決めることが最大の価値**: PJを追加するのは簡単。このスキルの価値は、飽和したポートフォリオから「今は手を出さない」を明示的に選ぶことにある
- **Area起点で考える**: 個別PJを見る前に、Area全体の健康状態を確認する。これにより「木を見て森を見ず」を防ぐ
- **現実的な配置を最優先する**: 理想的な計画より、実行可能な計画。キャパシティを超えた配置は計画ではなく願望
- **判断の理由を残す**: なぜ保留にしたか、なぜこの優先度にしたかの理由をセッションに記録する。次回のリバランスで「あの時なぜこう判断したか」がわかる
- **完了を祝う**: 棚卸しで完了と判定したPJは、成果を簡潔に振り返ってから閉じる。達成感がなければ次のPJへの活力が生まれない
