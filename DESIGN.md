# Personal Operating System 設計

> 最終更新: 2026-04-04
> ステータス: 初版。実践しながらブラッシュアップしていく

---

## 目的

**自分の思考の質・スピードを高めることで、意思決定の精度・スピードを上げて成果に繋げる**

具体的に実現したいこと:
- 過去に考えたことをいつでも取り出して活用できる（第二の脳）
- 毎回ゼロから考えるのではなく、蓄積された思考の上に積み上げられる
- LLMエージェントとの認識ズレを最小化し、すぐに本題に入れる

**なぜ「PKM」ではなく「Personal OS」か:**
- PKM（Personal Knowledge Management）は知識の管理に閉じた概念
- このシステムは知識管理だけでなく、思考モードの切替（エージェント）、実行管理（プロジェクト）、外部サービス連携を含む統合系
- 「自分の仕事の回し方全体」を包含するOSと捉える方が自然

## 全体アーキテクチャ

### 設計思想: 疎結合なシステム群 + エージェント

```
                 エージェント（一緒に考える実体）
                /      |       \        \
           参照↓     参照↓     参照↓     参照↓
      思考空間     ナレッジ    タスク管理   チーム情報・
     (自分の思考)   ベース                 デザイン等
                  (外部知識)
```

**なぜこの構造か:**
- エージェントは外部情報によって振る舞いが変わる。同じ質問でも知っている情報が違えば回答が変わる
- だからシステム間の境界を明確にし、疎結合に保つ
- 各システムの中身の整理は各システムに任せる。設計すべきは「エージェントがどこに何を見に行くか」のマッピング（= sources.md）

### システムの役割分担

| システム | 役割 | 公開性 | 管理形態 |
|----------|------|--------|----------|
| **エージェント定義** | 「どう考えるか」を決める | パブリック可 | Gitリポジトリ |
| **思考空間** | 自分の思考の外部保存。「考える場所」 | **プライベート** | Obsidian vault |
| **ナレッジベース** | ベストプラクティス・事例・最新情報。「参照する場所」 | パブリック可 | Obsidian vault |
| **外部サービス** | 実務情報（タスク、チーム情報、デザイン等） | 個別 | 各サービス |

**なぜ思考空間とナレッジベースを分けるか:**
- 性質が異なる。自分の思考は文脈依存で主観的、外部知識は文脈非依存で客観的
- 相互参照の方向: ナレッジベースを参照しながら、思考空間で思考し意思決定する
- 公開可能性の境界が異なる。外部知識はパブリックにできるが、自分の思考は本質的にプライベート

## 情報管理の基本原則: raw → compiled

すべての情報管理システムに共通する基本パターン。

```
raw（素材・記録）──→ compiled（構造化・統合された知識）
```

- **raw**: 時系列で蓄積される一次情報。セッションログ、記事、メモ、リサーチ結果など
- **compiled**: rawから構造化・統合された参照用ドキュメント。テーマ別の現状認識、体系化されたナレッジなど

**なぜこのパターンが必要か:**
- rawは時系列で蓄積されるため、テーマが散在する。同じ課題について2月と3月に考えていても、記録は別々
- compiledがないと、毎回過去の記録を探して読み直す必要がある（= ゼロから考えている感覚の正体）
- compiledがあれば、エージェントがそれを読み込むだけで最新の認識状態から会話を始められる（認識ズレ解消）

**各システムへの適用:**

| システム | raw | compiled |
|----------|-----|----------|
| **思考空間** | セッションログ | テーマ別の現状認識 |
| **ナレッジベース** | 記事、論文、リサーチ結果など | 体系化されたナレッジ、ベストプラクティス集 |

### compiled ドキュメントの構造

compiledドキュメントに含めるべき要素:

1. **現状の事実** — 確定事項、数字、構成
2. **構造** — 因果関係、分類、優先順位
3. **判断の履歴** — いつ何をどう決めたか、その理由
4. **未決事項・オープンな問い** — まだ答えが出ていないこと
5. **ソースへのリンク** — 詳細を掘りたい時にrawに戻れるように

**なぜこの構造か:**
- 1-2があれば「今どうなっているか」をすぐ把握できる
- 3があれば「なぜこうなったか」の文脈が失われない
- 4があれば「次に何を考えるべきか」が明確
- 5があれば深掘りしたい時にrawに戻れる（compiledは要約なので情報は落ちている）

### compiledの生成・更新

**compiledはrawからのみ生成する。外部情報（タスク管理やチームツール等）は含めない。**

**なぜrawに限定するか:**
- compiledの目的は「自分の思考の蓄積」。外部の最新状況ではない
- 外部情報を含めるとすぐ古くなり、鮮度管理が複雑になる
- 外部の最新情報はセッション時にエージェントが都度取りに行けばいい

**実行場所:** 各システム内のスキルとして実行する。エージェントから横断的に呼ぶのではなく、各システムのセッション内で完結する。

**なぜ各システム内で実行するか:**
- compiledの素材（raw）はそのシステム内に閉じている
- システムごとにcompileの仕方が異なる（思考空間はセッションログの統合、ナレッジベースは記事の体系化）
- エージェントは各システムのcompiledが整っている前提で読みに行く。不足があればフィードバックする

**更新タイミング:**
- **セッション後**: そのテーマのcompiledに差分があれば更新（エージェントが提案、自分が承認）
- **状況が変わった時**: 採用が決まった、方針が変わった等。気づいた時にcompiledを修正
- **週次レビュー時**: 未更新のcompiledがないかチェック

**狙い:** メンテナンス対象をcompiledに絞ることで負荷を最小化する。rawは今まで通り書くだけでいい。

## 思考空間の内部構造

### 2つの層 + 実行管理

| 層 | 性質 | 比喩 |
|----|------|------|
| **思考のフロー** | 一時的・探索的。セッションで考えたこと | 川の流れ |
| **思考のストック** | 永続的・統合的。テーマ別の現状認識 | 湖 |

**なぜこの2層か:**
- 思考にはフロー（探索的に考える過程）とストック（統合された認識）の2つのモードがある
- フローだけだと毎回ゼロから。ストックだけだと思考過程が失われる。両方必要

### 実行管理の位置づけ

実行管理は思考とは性質が異なる。個人プロジェクトとチームプロジェクトが混在しており、情報の置き場所も分かれる:

| 種類 | 置き場所 |
|------|----------|
| 個人プロジェクト | 思考空間内 |
| チームプロジェクト | 共有システム |
| 成果物（共有が必要なもの） | 共有システム |

**なぜ集約せず散らばったままにするか:**
- 成果物は他メンバーと共有する必要がある。共有システムに置く方が自然
- 個人の実行管理はローカルで十分
- 情報が散らばっていても、何がどこにあるかをエージェントが把握していれば十分（= sources.mdの役割）
- 集約より**マッピングの正確性**を保つ方が、疎結合の設計思想と一致する

### 層の間の流れ

```
フロー（考える）──→ ストック（統合する）──→ 意思決定 ──→ 実行
                        ↑                                  │
                        └──────── 学び・新たな問い ─────────┘
```

- **フロー → ストック**: セッションで考えたことを、テーマ別に統合する（raw → compiled）。これが核心プロセス
- **ストック → 実行**: 蓄積された認識から意思決定が生まれ、目標や施策に落ちる。成果物は必要に応じて共有システムへ
- **実行 → フロー**: 実行の中で生まれた問いや学びが、新たなセッションのテーマになる

**ストックが思考の中心地。** ここに自分の現状認識が蓄積されていて、そこから意思決定が生まれる。

## エージェントの位置づけ

### エージェントとは

一緒に考える実体。道具ではなく、思考のパートナー。

### エージェントの振る舞いを決める3つの要素

1. **思考の憲法**: 「何を大切にして考えるか」。すべてのエージェント・すべてのセッションに通底する思考原則
2. **エージェント定義**: 「どう考えるか」。コーチは問いで導き、参謀は率直にフィードバックする
3. **参照する外部情報**: 「何を見て考えるか」。同じ質問でも見ている情報が違えば回答が変わる

```
思考の憲法
  └→ すべてのエージェントの判断基準・価値観を規定
       │
  エージェント定義
    └→ 憲法の上に、モードごとの振る舞いを定義
         │
    参照する外部情報（sources.md）
      └→ セッションごとに読み込む情報で出力が変わる
```

**なぜ3層か:**
- 憲法は「ブレてはいけないもの」。エージェントのモードが変わっても、思考の誠実性や構造化の原則は変わらない
- エージェント定義は「モードごとに変わるもの」。コーチと参謀では対話の仕方が違う
- 外部情報は「セッションごとに変わるもの」。同じエージェントでも見ている情報で出力が変わる
- この3層を分離することで、それぞれ独立に改善できる

### エージェントの情報参照パターン

sources.mdで定義。セッション開始時にどの情報を読み込むかを頻度ガイドラインで制御:
- 日次: 週次目標、デイリーノート、マイルストーン
- 週次: 月次目標、プロジェクト、責任領域
- 必要時: メンバー情報、過去の意思決定記録、ナレッジベースのノート

**狙い:** compiledドキュメントが最新であれば、エージェントはそれを読み込むだけで認識が合う。「毎回説明するのが面倒で自分で考えちゃう」問題を構造的に解消する。

## ナレッジベース

思考空間の外にある参照ライブラリ。ベストプラクティス、他社事例、最新情報が構造化されている場所。Obsidian vaultとして管理。

### 構造

| 領域 | 役割 | raw/compiled |
|------|------|-------------|
| Inbox | 未整理の素材。取り込んだがまだ処理していないもの | raw |
| Clippings | 記事・Webクリップ | raw |
| Reports | リサーチレポート | raw → compiled |
| Notes | 整理されたエバーグリーンノート | compiled |
| Books | 書籍メモ | raw → compiled |

### raw → compiledの適用

- **raw**: Inbox、Clippings、個別の記事メモ、リサーチ結果
- **compiled**: Notes内のテーマ別に体系化されたナレッジ（MOC = Map of Contentで構造化）

思考空間との関係:
- ナレッジベース → 思考空間: 思考の材料として参照する
- 思考空間 → ナレッジベース: 実践から得た汎用的な学びを還元する（将来的に）

## 自己改善プロセス

Personal OS自体を継続的に改善するための仕組み。

### 改善シグナルの蓄積

セッション中に気づいた改善点・改善候補を `context/improvements.md` に貯める。

**シグナルの例:**
- エージェントの指示が不明確で意図と違う出力が出た
- compiledが古くて認識ズレが起きた
- ナレッジベースに必要な知識がなかった
- スキルの使い勝手が悪かった
- sources.mdのマッピングに漏れがあった

**なぜagentsに集約するか:**
- 改善を実行する時にまとめて見たい
- 各システムの改善もエージェントとのセッションで実行するため、起点はここ
- 将来的には各システムに改善シグナルを残し、各システムが自己修復する形に進化させる

### 改善の実行

改善シグナルを見て何をどう改善するかは、自分のタイミングで手動で判断・実行する。

**なぜ自動化しないか:**
- システムの安定性を保つ。自動改善は意図しない変更を生むリスクがある
- 改善の優先順位は文脈によって変わる。人間が判断すべき
- まずは「貯める」仕組みだけで十分。運用が回ってから自動化を検討する

## 設計原則

1. **思考と知識を分ける** — 自分の思考（主観的・プライベート）と外部知識（客観的・パブリック可）は別システムに置く
2. **rawとcompiledを分ける** — 一次情報（フロー）と構造化された知識（ストック）は別の層。すべての情報管理システムに適用する基本パターン
3. **疎結合に保つ** — エージェント、思考、知識、外部サービスはそれぞれ独立。接続はマッピング（sources.md）で管理
4. **compiledを中心に据える** — メンテナンス対象をcompiledに絞り、ここを最新に保つことで全体が回る
5. **エージェントの認識 = compiledの品質** — エージェントのアウトプットはコンテキストの品質に制約される。compiledの質がエージェントの質を決める
